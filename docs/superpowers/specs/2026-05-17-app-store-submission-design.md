**Date:** 2026-05-17
**Status:** Approved

## Goal

Get both Amana apps (Agent + Principal) live on TestFlight and Google Play Store closed testing, with proper EAS build pipeline, branded visual assets, and complete store listings.

## Scope

**In scope:**
- EAS build configuration for both apps (Agent + Principal)
- App icons and splash screens (designed from scratch)

## Sub-project 1: EAS Build Pipeline

### Architecture

Each app (`apps/agent/`, `apps/principal/`) gets its own `eas.json` with three build profiles:

| Profile | Distribution | Android | iOS | Purpose |
|---|---|---|---|---|
| `development` | internal | dev client | dev client | Expo dev client for local development |
| `preview` | internal | APK | IPA | Easy sideload for internal testers |
| `production` | store | AAB | IPA | Store submission |

### Credentials

**iOS:** EAS managed credentials. No existing App Store listing, so a fresh provisioning profile and distribution certificate are generated by EAS and stored in their credentials service.

**Android:** Play App Signing is almost certainly enabled (all apps uploaded after August 2021 use it by default). The plan's first step is to verify in Play Console → App signing. If enrolled: EAS generates a new upload key, Google handles app signing. If not enrolled: locate the original keystore before proceeding.

### Versioning

- Check Play Console for the current `versionName` and `versionCode` of each app before setting values in `app.json`.
- Set `version` (versionName) and `android.versionCode` in `app.json` to match or exceed what's currently on Play Store.
- Set `ios.buildNumber` to `"1"` (fresh App Store listing).
- Enable `autoIncrement: "buildNumber"` on the production profile so EAS increments `ios.buildNumber` and `android.versionCode` automatically on each production build. The semver `version` field is bumped manually when releasing new features.

### Files

- `apps/agent/eas.json` — EAS config for Agent
- `apps/principal/eas.json` — EAS config for Principal
- `apps/agent/app.json` — updated version, icon/splash paths, android.versionCode
- `apps/principal/app.json` — updated version, icon/splash paths, android.versionCode

## Sub-project 2: Visual Assets

### Brand System

Two-color identity — one color per app role:

| App | Role | Color | Hex |
|---|---|---|---|
| Principal | Money controller | Deep teal | `#0D7377` |
| Agent | Sub-wallet holder | Warm amber | `#E07B39` |

### Icons

Each app gets:
- `assets/icon.png` — 1024×1024 PNG, no alpha channel (iOS + Android legacy)
- `assets/adaptive-icon.png` — 1024×1024 foreground with padding (Android adaptive icon)

**Principal icon:** Teal background, white shield mark — conveys control and security.
**Agent icon:** Amber background, white wallet/card mark — conveys the recipient role.

Designed using the `frontend-design` skill, exported as PNG.

### Splash Screens

- Solid background in each app's brand color
- Centered white logo mark (same as icon, ~40% viewport width)
- Configured via `expo-splash-screen` plugin in `app.json`

File: `assets/splash.png` — 1284×2778 PNG per app (covers largest iPhone screen)

### Screenshots

Captured from running apps in iOS Simulator and Android Emulator — not designed mockups.

**Screens to capture per app:**

| App | Screens |
|---|---|
| Principal | Login/OTP, Home (wallet overview), Agent management |
| Agent | Login/OTP, Transaction list, Transaction detail |

**Store requirements:**

| Store | Requirement | Size |
|---|---|---|
| Play Store | 2× phone screenshots | Min 1080px, 16:9 or 9:16 |
| Play Store | 1× feature graphic | 1024×500 |
| App Store | 3× iPhone 6.5" screenshots | 1284×2778 |

## Sub-project 3: Store Submission

### Privacy Policy

A single static HTML page covering both apps, hosted on GitHub Pages at `https://alexander77063.github.io/amana/privacy` (served from the `gh-pages` branch, `privacy.html` at repo root of that branch). Content:
- Data collected: phone number, transaction history, location (Agent only)
- How data is used: wallet operations, fraud prevention
- Third parties: Anchor API (payment processing)
- Data retention and deletion
- Contact email: `alexanderadegbola@gmail.com`

### App Store Connect

Two app entries:
- **Amana Agent** — bundle ID `com.amana.agent`, category: Finance
- **Amana Principal** — bundle ID `com.amana.principal`, category: Finance

Each entry requires: app name, primary language (English), privacy policy URL, age rating (4+), screenshots, and short description before TestFlight external testing can be enabled.

### Play Console

The existing closed testing track already has a build. Complete the store listing:
- Short description (80 chars max)
- Full description (4000 chars max)
- Screenshots (phone + feature graphic)
- Content rating questionnaire (IARC)
- Privacy policy URL
- Upload new AAB from EAS production build

### Submission Flow

1. Run `eas build --platform all --profile production` for each app
2. Run `eas submit` (or upload manually via Transporter for iOS, Play Console for Android)
3. TestFlight: build processes (~30 min) → add to external test group → share invite link
4. Play Store: upload AAB to closed testing track → share invite link

### Deliverables

- TestFlight invite link for Amana Agent
- TestFlight invite link for Amana Principal
- Play Store closed testing invite link for Amana Agent
- Play Store closed testing invite link for Amana Principal
